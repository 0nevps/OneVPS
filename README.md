# OneVPS

[中文版](README_CN.md)

One-click proxy node deployment script for VPS, powered by [sing-box](https://github.com/SagerNet/sing-box).

Supported protocols:

- **VLESS + Reality** — Direct connection, no domain/certificate needed, TLS fingerprint camouflage
- **VLESS + WebSocket + CF CDN** — Route through Cloudflare CDN, hide real IP
- **SOCKS5** — Authenticated SOCKS5 proxy inbound

Optional enhancements:

- 🟠 **Cloudflare CDN** — Route VLESS through CF (auto-switches to WS transport)
- 🧦 **SOCKS5 Outbound** — Chain an authenticated SOCKS5 exit for any node, VPS acts as relay
- 🚀 **BBR Acceleration** — One-click kernel BBR TCP congestion control for better throughput

---

## Quick Start

One-liner (remote):

```bash
bash <(curl -fsSL https://raw.githubusercontent.com/0nevps/OneVPS/main/onevps.sh)
```

Or download and run locally:

```bash
sudo bash onevps.sh
```

The script runs environment checks (root / architecture / package manager / systemd) before entering the menu.

First time: run `1` to install sing-box, then `2`/`3` to add nodes.

## Protocols & Modes

| Scenario | Protocol | Transport | Domain | Certificate |
|----------|----------|-----------|--------|-------------|
| VLESS direct | VLESS + Reality | TCP direct | Not needed | Not needed (Reality camouflage) |
| VLESS via CF | VLESS + WS | WebSocket via CF | Required | Self-signed (CF terminates TLS) |
| SOCKS5 | SOCKS5 | TCP | Not needed | Not needed |

### VLESS + Reality

- Direct to VPS, no domain or certificate required
- Camouflage target must support TLS 1.3 + H2 (presets: microsoft.com, apple.com, etc.)
- Client uses `flow: xtls-rprx-vision`
- Share link format: `vless://...?security=reality&fp=chrome&pbk=...&sid=...`

### VLESS + WS + CF CDN

- Share link format: `vless://...?security=tls&type=ws&host=...&path=...`

#### Cloudflare Setup

1. **Host domain on CF** — Point domain NS records to Cloudflare (change at registrar)
2. **Add DNS record** — CF Dashboard → DNS → Add A record:
   - Name: your domain (e.g. `proxy.example.com`)
   - Content: VPS real IP
   - Proxy status: **Proxied** (orange cloud ☁️ on)
3. **SSL/TLS encryption mode** — CF Dashboard → SSL/TLS → Overview → Select **Full**
   - ⚠️ Do NOT select "Full (Strict)" — this script uses self-signed certs, strict mode will reject
   - ⚠️ Do NOT select "Flexible" — flexible mode sends plaintext HTTP to origin, insecure
4. **WebSocket** — Enabled by default on CF, no extra config needed
5. **Origin port** — CF free plan only proxies these HTTPS ports (script enforces this):
   ```
   443  2053  2083  2087  2096  8443
   ```

#### Notes

- **Propagation time**: After enabling proxy, usually takes minutes, occasionally hours
- **IP leak**: Before enabling orange cloud, ensure no other DNS records (MX, unproxied subdomains) expose VPS IP
- **CF free plan limits**: No hard WebSocket connection limit, but high traffic may trigger CF security rules (Under Attack mode, Rate Limiting)
- **Client SNI/Host**: Must use domain (not IP), share links generated by the script handle this automatically

> **Why self-signed with CF?** CF's orange cloud intercepts ACME HTTP-01 validation, preventing real cert issuance.
> Self-signed + CF "Full" mode keeps VPS↔CF traffic encrypted.

### SOCKS5

- Authenticated SOCKS5 proxy with username/password
- No domain or certificate needed, lightest node type
- Optional SOCKS5 outbound chaining (client → VPS SOCKS5 inbound → remote SOCKS5 outbound)
- Share link format: `socks5://user:pass@ip:port#name`

## SOCKS5 Outbound

When adding a node or in "Manage Nodes", you can attach a SOCKS5 exit:

```
SOCKS5 server address: residential.example.com
SOCKS5 port: 1080
Username: user
Password: ******
```

Once attached, **all traffic** (including DNS) for that node exits via SOCKS5. VPS only acts as a relay.

> ⚠️ No failover: if the SOCKS5 exit goes down, the node becomes unavailable.

## File Locations

| Path | Purpose |
|------|---------|
| `/usr/local/bin/sing-box` | Core binary |
| `/etc/sing-box/nodes.json` | Node metadata (single source of truth) |
| `/etc/sing-box/config.json` | sing-box runtime config (auto-generated from nodes.json) |
| `/etc/sing-box/certs/` | Self-signed certificates (not used by Reality) |
| `/etc/systemd/system/sing-box.service` | systemd service |

Manual service management:

```bash
systemctl status sing-box
systemctl restart sing-box
journalctl -u sing-box -f
```

## BBR Acceleration

Menu `8` enables Google BBR TCP congestion control (requires kernel ≥ 4.9).

Writes to `/etc/sysctl.d/99-bbr.conf`, sets `fq` queue discipline + `bbr` congestion control. Persists across reboots.

Status bar shows current TCP congestion algorithm (e.g. `TCP:bbr`).

Menu `9` — removes binary, config, all nodes, and certificates.
